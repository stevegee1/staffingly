# StaffVerify Project Conventions

## Build and Run Commands

### Backend
- `npm start` (in `backend/`): Start production server
- `npm run dev` (in `backend/`): Start development server with nodemon
- `npm run lint` (in `backend/`): Run ESLint
- `npm run prisma:generate` (in `backend/`): Generate Prisma client
- `npm run prisma:push` (in `backend/`): Push database schema
- `npm run prisma:migrate` (in `backend/`): Create database migration
- `npm run prisma:seed` (in `backend/`): Seed database

### Frontend
- `npm run dev` (in `frontend/`): Start development server
- `npm run build` (in `frontend/`): Build for production
- `npm run preview` (in `frontend/`): Preview production build
- `npm run lint` (in `frontend/`): Run ESLint

## General Coding Principles

### Code Readability
- Write self-documenting code that reads like prose — avoid unnecessary comments
- Name variables, functions, components, and classes to express intent clearly
- Structure code to flow logically from top to bottom
- Group related logic together; separate concerns into distinct sections
- Use comments only to explain _why_ something complex or counter-intuitive is being done

### Core Principles
- **SOLID**: Follow Single Responsibility, Open/Closed, Liskov Substitution, Interface/Prop Segregation, and Dependency Inversion.
- **KISS**: Prefer simple solutions over clever ones. Avoid premature abstraction.
- **DRY (with judgment)**: Extract repeated logic only when the pattern is stable (three occurrences threshold).

## Backend Specifics

### Architecture & File Organization
- **Layered Architecture**: Routes → Controllers → Services → Data Layer (Prisma).
- Keep files under 250 lines; extract services or utilities when approaching this limit.
- Decouple business logic from framework-specific code (e.g., service layer should not know about `req` or `res`).
- **File Structure**: `src/routes/`, `src/controllers/`, `src/services/`, `src/middleware/`, `src/lib/`.
- **Order**: imports → constants → types/interfaces → class/functions → exports.

### Base44 Independence
- **No Base44 Dependency**: The application MUST NOT depend on the @base44/sdk or any Base44-specific infrastructure.

## Frontend Specifics

### Architecture & File Organization
- Keep files under 250 lines; extract components or utilities when approaching this limit.
- One component per file; co-locate related hooks and helpers only if small.
- **Order**: imports → constants → types → component → exports.

### Performance
- Memoize expensive computations with `useMemo`.
- Prevent unnecessary re-renders with `React.memo` and `useCallback` where impactful.
- Lazy load routes and heavy components.
- Avoid inline object/array creation in JSX props.

## Shared Standards

### Imports Organization
1. Framework/Built-in modules (e.g., `react`, `fs`, `path`)
2. External dependencies (e.g., `express`, `prisma`, third-party packages)
3. Internal/absolute imports (e.g., `@/components`, `@/services`, `@/lib`)
4. Relative imports from parent directories (`../`)
5. Relative imports from the same directory (`./`)
6. Styles and static assets (Frontend specific)

### Naming Conventions
- **Components/Classes/Models**: PascalCase (`UserProfile`, `UserService`)
- **Functions/Variables/Hooks**: camelCase (`getUserById`, `useAuth`)
- **Constants**: SCREAMING_SNAKE_CASE (`MAX_RETRY_COUNT`)
- **Files**: Match the primary export name or follow directory conventions (e.g., `user.controller.ts`)

### Security
- Sanitize and validate all user inputs at system boundaries.
- Develop proper CORS configuration (Backend).
- Never expose sensitive data or internal error details in production/logs.
- Use environment variables for API keys, database URLs, and secrets.
- Avoid `dangerouslySetInnerHTML` unless absolutely necessary and sanitized.

### Error Handling
- Use global error handling patterns where applicable.
- Handle errors at appropriate boundaries.
- Provide meaningful and standardized error responses/states to the user/client.
- Log errors with sufficient context for debugging, but sanitize sensitive info.
